Core Flow

Auto-Remediation Pipeline

When a sign-in failure event is detected that meets remediation criteria, the system executes this 5-step pipeline automatically — no human intervention required.
Event Detection
Event Hub consumer receives sign-in failure event. Error code is checked against remediation codes: 50053 (Account Locked), 50057 (Account Disabled), 50055 (Password Expired). For bad password events (50126), failures are counted — threshold breach (default: 5 in 30 min) triggers remediation.
User Lookup
Atomicwork /api/v1/users/{email}/email-lookup resolves the user's full name, phone number, and requester ID. These are used for SMS delivery and ticket creation.
Password Reset + Account Unlock
Microsoft Graph API PATCH /users/{upn} sets a cryptographically random 12-character temporary password (uppercase + lowercase + digit + special) with forceChangePasswordNextSignIn: true, and re-enables the account.
SMS Notification
Twilio sends an SMS to the user's phone number with the temporary password and a link to myaccount.microsoft.com to change it immediately. Skipped if no phone number is on file.
Ticket Creation + Auto-Resolve
Atomicwork POST /api/v1/requests/create creates a high-priority incident ticket with full details. The ticket is immediately resolved via PATCH /api/v1/requests/{id} since remediation was successful — creating a complete audit trail.
Detection Logic

When Remediation Triggers

Two distinct trigger paths lead to auto-remediation.
🚨

Instant Trigger — Critical Error Codes

If the sign-in failure error code is one of the critical codes (Account Locked, Account Disabled, Password Expired), remediation fires immediately on the first event. No counting, no waiting.

50053 — Locked 50057 — Disabled 50055 — Expired
📊

Threshold Trigger — Bad Password Accumulation

Bad password attempts (error code 50126) are counted per user within a rolling 30-minute window. When the count reaches the configured threshold (default: 5), remediation is triggered. The window slides — old failures age out.

50126 — Bad Password Default Threshold: 5
🛡️

Deduplication

Once a user has been remediated, they're added to a dedup set. Subsequent events for the same user are processed and displayed in the live feed, but no duplicate remediation runs. This prevents password thrashing during an active incident.

In-Memory Dedup Set
Frontend

Web UI — 5 Pages

React single-page application served from a single HTML file with Tailwind CSS and dark glass-morphism theme.
📊

Dashboard

Overview of recent sign-in failures fetched from Graph API. Shows error codes, user names, IPs, locations, and timestamps. Critical events are highlighted.

📡

Live Feed

Real-time event stream via WebSocket. Events appear as they flow from Event Hub. Start/Stop controls for the consumer. Auto-remediation results broadcast live.

👥

User Directory

Lists all users from both Azure AD and Atomicwork. Shows name, email, phone, department, and status. Phone numbers enriched via Atomicwork email-lookup API.

🔧

Remediation

Manual remediation trigger by email. Executes the full 5-step pipeline and shows step-by-step results. Also displays remediation history log.

⚙️

Settings

Encrypted credential management for all 4 services. Secrets are masked in the UI. Test Connection buttons verify each integration. Lockout threshold configuration.

Security

Security Architecture

🔒

Fernet Encryption (AES-128-CBC)

All credentials (Azure secrets, API keys, Twilio tokens) are encrypted at rest using Fernet symmetric encryption. The encryption key is stored with file permissions 600.

🎭

Secret Masking in UI

The GET /api/config endpoint returns masked values (first 6 + last 4 chars visible). Secrets never appear in browser DevTools network tab.

🔐

MSAL Confidential Client

Azure auth uses MSAL ConfidentialClientApplication with client credentials flow. Tokens are cached in memory with TTL-based refresh.

🔑

Temp Password Generation

Passwords generated using Python's cryptographic secrets module. 12 chars with guaranteed uppercase, lowercase, digit, and special character.

🌐

Network Restriction

Azure App Service supports IP-based access restrictions. The deploy script provides a command to lock down to specific IP ranges (internal only).

🚫

Masked Value Rejection

When saving config, any field containing the mask character (•) is silently ignored — preventing accidental overwrite with masked placeholder text.

Reads sign-in audit logs, lists users with phone numbers, resets passwords, and re-enables accounts. Requires an App Registration with client secret.

Receives real-time sign-in log events via Azure AD Diagnostic Settings. The app connects as a consumer on the $Default consumer group, reading from @latest position.
